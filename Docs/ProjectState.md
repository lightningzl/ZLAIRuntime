不同文档分别维护一种事实，发生冲突时按下表处理：

| 文档 | 唯一职责 | 冲突处理 |
| --- | --- | --- |
| [CurrentMilestone.md](./CurrentMilestone.md) | 当前范围、明确不做、验收标准、完成定义 | 范围和验收以此为准 |
| [Protocol.md](./Protocol.md) | UE 与 Python Service 的通信契约 | 字段、状态码和兼容性以此为准 |
| [Architecture.md](./Architecture.md) | 模块职责、边界和依赖方向 | 设计边界以此为准 |
| [TaskBoard.md](./TaskBoard.md) | 工作包、状态、依赖和任务完成条件 | 执行状态以此为准 |
| `ProjectState.md` | 当前里程碑的汇总快照 | 必须从上述文档和验证事实同步，不得覆盖它们 |
| [DecisionLog.md](./DecisionLog.md) | 已接受的重要决策及原因 | 设计取舍的原因以此为准 |

代码和测试是实现与行为的最终证据；文档状态与实际实现不一致时，应将状态降为“未验证”并修正文档，不能仅修改描述来宣称完成。

## Project State 规则

### 1. 允许的里程碑状态

| 状态 | 使用条件 |
| --- | --- |
| `已规划` | 范围与任务已定义，但尚无任务进入实施 |
| `进行中` | 至少一个当前里程碑任务正在实施 |
| `受阻` | 关键路径无法继续，且没有可推进的同范围替代任务 |
| `验收中` | 实现任务完成，正在执行里程碑级验证 |
| `已完成` | Current Milestone 的全部验收项均有通过证据，并满足完成定义 |

Project State 不使用百分比估算主观进度。验收进度只统计 [CurrentMilestone.md](./CurrentMilestone.md) 中已有明确证据的验收项。

### 2. 必须更新的触发条件

发生以下任一事件时，必须在同一工作变更中更新本文件：

- Task Board 中任务进入 `进行中`、`受阻`、`待验收` 或 `已完成`。
- 当前活动任务、下一候选任务或关键依赖发生变化。
- 新增、解除或改变会影响当前里程碑的阻塞。
- 某项里程碑验收首次获得证据，或原有证据失效。
- 当前里程碑进入 `验收中` 或 `已完成`。
- 用户批准范围、协议或架构变化，导致当前快照发生变化。

仅修改注释、命名、格式或其他不改变项目状态的内容时，不更新本文件日期。

### 3. 状态更新要求

- `最后更新` 使用 `YYYY-MM-DD`，表示内容最后一次发生实质变化的日期。
- `当前活动任务` 使用 Task Board ID；没有活动任务时明确写“无”。
- `下一候选任务` 只能列出依赖已满足或可独立启动的当前里程碑任务。
- 阻塞项必须写明原因、受影响任务和解除条件；不能只写“等待处理”。
- 已完成内容必须有代码、测试、运行结果或明确文档产物作为证据。
- 无法验证或证据不完整时使用“未验证”，不得推定成功。
- 不记录个人临时笔记、详细实现步骤、完整日志或未来里程碑愿望清单。

### 4. 与 Task Board 的同步

- Task Board 是任务状态的事实来源，Project State 只汇总活动任务、下一步、阻塞和验收进度。
- 两份文档出现不一致时，先根据实现和验证证据修正 Task Board，再同步 Project State。
- Task Board 的任务进入 `进行中` 时，里程碑从 `已规划` 变为 `进行中`。
- 所有实现任务完成后，里程碑进入 `验收中`；不能直接跳到 `已完成`。
- `M1-07` 完成且全部验收通过后，里程碑才能标记为 `已完成`。

### 5. 范围与决策边界

- 本文件不得新增 [CurrentMilestone.md](./CurrentMilestone.md) 未授权的功能。
- 本文件不得重新定义 [Protocol.md](./Protocol.md) 中的字段、状态码或语义。
- 新的架构或协作决策按 [DecisionLogRules.md](./DecisionLogRules.md) 判断是否写入 Decision Log。
- 协议修改必须先向用户说明并获得确认，随后同步协议、任务板和项目状态。

### 6. Agent 开始与结束流程

开始任务时：

1. 阅读 `AGENTS.md` 指定的必读文档。
2. 核对当前里程碑、活动任务、阻塞和下一候选任务。
3. 检查代码与 Task Board 是否一致；不一致时先报告并修正状态。
4. 将所选任务改为 `进行中`，同步本文件后再实施。

结束任务时：

1. 记录实际完成内容和验证结果。
2. 更新 Task Board 状态。
3. 更新本文件的活动任务、下一候选任务、阻塞和验收进度。
4. 更新适用的 UE、Python 或 Decision Log 文档。
5. 按 [GitWorkflow.md](./GitWorkflow.md) 检查 diff；未经用户明确要求不创建 commit。
