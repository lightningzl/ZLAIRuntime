# Documentation Rules

## 目标

每份文档只维护一种事实。需要了解同一事项时应通过链接追溯唯一来源，而不是在多份文档复制完整描述。

## 文档职责与唯一来源

| 文档 | 唯一职责 | 不负责 |
| --- | --- | --- |
| `ProjectOverview.md` | 项目定位、长期目标、技术栈和演进路线 | 当前任务、验收细节、实现状态 |
| `Architecture.md` | 系统模块、职责边界和依赖方向 | 文件清单、配置默认值、任务状态 |
| `Protocol.md` | UE 与 Python 的通信字段、语义、状态码和兼容性 | Provider SDK、Prompt、实现步骤 |
| `CurrentMilestone.md` | 当前范围、明确不做、验收标准和完成定义 | 任务状态、验证日志、历史里程碑 |
| `TaskBoard.md` | 当前里程碑的工作包、依赖、状态和完成条件 | 重新定义范围、协议或完整验收标准 |
| `ProjectState.md` | 当前里程碑的简洁快照：活动任务、下一步、阻塞和已验证进度 | 流程规则、架构、协议、任务明细 |
| `Validation/MilestoneNValidation.md` | 已执行验证的命令、环境、结果和证据 | 重新定义验收标准或任务计划 |
| `UEClasses.md` | AI Runtime 相关 UE 类型、职责和依赖 | 系统级架构和任务状态 |
| `PythonModules.md` | Python 模块、内部边界、运行配置和实现级验证要求 | UE 类型和 HTTP 协议语义 |
| `CodingStandards.md` | 代码风格、实现约束和通用测试约束 | 当前范围、架构选择和 Git 流程 |
| `GitWorkflow.md` | 分支、暂存、提交和提交前验证流程 | 模块职责和当前任务状态 |
| `DecisionLog.md` | 已接受的重要架构和取舍，以及“为什么” | 任务细节、临时实现笔记和状态快照 |
| `Milestones/MilestoneN.md` | 已完成里程碑的范围定稿归档 | 当前范围或状态 |
| `AGENTS.md` | Agent 进入仓库后的必读清单和强制入口 | 复制各文档的完整规则 |

## 冲突处理

按事实类型确定来源，而不是按文档更新时间覆盖：

- 通信格式、字段和状态码以 `Protocol.md` 为准。
- 当前范围、明确不做和验收标准以 `CurrentMilestone.md` 为准。
- 任务状态和依赖以 `TaskBoard.md` 为准。
- 当前活动任务、阻塞和验收进度以 `ProjectState.md` 为准；它必须从前述来源同步，不得反向覆盖。
- UE/Python 类型或模块职责以对应模块文档为准；系统边界发生冲突时以 `Architecture.md` 为准并更新模块文档。
- 已执行行为以代码、测试和运行结果为准。若证据与文档不一致，先将状态标为“未验证”，再修正文档。
- `DecisionLog.md` 解释已接受决策的原因，不得单独改变协议、范围或状态。

## 编写规则

- 文档使用链接引用唯一来源，不复制其完整表格、规则或验收文字。
- 当前范围以外的功能只能记录在项目路线或 Decision Log 中，不得加入当前任务队列。
- 不记录密钥、完整玩家输入、内部路径、完整堆栈、原始 Provider 异常或临时调试日志。
- 配置示例只能使用占位值；本地 `.env` 不得提交。
- 新增或改名文档时，更新所有入站链接；文档内 JSON 示例必须是有效 JSON。
- 文档只记录已确认的事实。未执行的验证明确标记“未验证”。

## 里程碑与验收

- 当前里程碑的每项验收标准必须有稳定 ID，例如 `M2-A01`。
- `CurrentMilestone.md` 是验收标准的唯一正文来源。
- `TaskBoard.md` 的最终验收任务只引用验收 ID，不复制标准。
- 验收记录只列 ID、状态和证据；详细标准通过链接回 `CurrentMilestone.md`。
- 一个任务只有在其完成条件满足且验证证据可复查时才能标记为 `已完成`。
- 所有实现任务完成后，里程碑先进入 `验收中`；最终验收任务和全部验收 ID 通过后才能标记 `已完成`。

## 状态更新

以下变化必须在同一工作变更中同步 `TaskBoard.md` 与 `ProjectState.md`：

- 任务进入 `进行中`、`受阻`、`待验收` 或 `已完成`。
- 活动任务、下一候选任务、阻塞或已验证验收项发生变化。
- 当前里程碑进入 `验收中` 或 `已完成`。

阻塞必须写明原因、受影响任务和解除条件。没有活动任务时明确写“无”。

## 协议、架构与模块变更

- 修改协议字段、类型、语义、状态码或兼容规则前，必须先向用户说明并获得确认；确认后同步 `Protocol.md`、`CurrentMilestone.md`、`TaskBoard.md` 和 `ProjectState.md`。
- 新增或改变系统模块边界、依赖方向或长期技术取舍时，同步更新 `Architecture.md`、相关模块文档，并按下方规则写入 `DecisionLog.md`。
- 改变 UE 类型职责或公开接口时，更新 `UEClasses.md`。
- 改变 Python 模块职责、内部接口、运行配置或依赖方向时，更新 `PythonModules.md`。

## Decision Log 规则

满足以下任一条件的决策必须记录到 `DecisionLog.md`：架构、长期可维护性、模块边界、技术选型、重要取舍或面试讲解价值。

不要记录函数名、变量名、临时调试、按键、日志类别或常规目录调整。无法判断时，不自动记录，应在交付中说明。

每条决策使用以下结构：

```md
## YYYY-MM-DD：简短标题

决定：
...

原因：
- ...

取舍：
- ...

状态：
已接受

重要性：
重要
```

## 归档规则

- 切换到新里程碑前，将完成时的 `CurrentMilestone.md` 定稿复制到 `Docs/Milestones/MilestoneN.md`。
- 归档文档只做历史追溯，不得随着当前协议、架构或实现变化而改写；必要的勘误必须说明原因和日期。
- 当前里程碑文档、任务板和项目状态应链接相关归档和验收记录，但不复制其内容。

## Agent 工作流程

开始实现前：

1. 按 `AGENTS.md` 读取必读文档和本规则。
2. 检查分支、工作区、当前范围、任务状态和阻塞。
3. 将选定任务更新为 `进行中`，同步 `ProjectState.md`，再修改代码。

结束任务时：

1. 记录实际结果和验证证据。
2. 更新任务状态与项目快照。
3. 更新受影响的协议、架构、模块或决策文档。
4. 按 `GitWorkflow.md` 检查 diff；未获明确要求不得创建提交、推送、变基、合并或 PR。

## 文档验证

文档变更至少检查：相对链接、JSON 示例、术语和唯一来源是否一致，以及 `git diff --check`。仅文档变更不要求运行代码测试；未执行测试必须如实说明。
